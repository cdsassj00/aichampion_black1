01모델 선택 이유

Gemma 4 E4B를 선택한 가장 큰 이유는 모바일·엣지 디바이스에서 실행 가능한 소형 모델임에도 멀티모달(텍스트·이미지·오디오) 입력을 동시에 지원하는 점입니다. 공공기관 업무 환경에서는 민감한 데이터를 외부 클라우드로 전송하지 않고 로컬 환경에서 처리해야 하는 경우가 많은데, E4B는 온프레미스·온디바이스 배포가 가능하면서도 문서 이미지 OCR, 음성 입력(ASR), 행정 문서 요약·질의응답 등 다양한 공공 업무에 적용할 수 있습니다.

또한 Gemma 4는 2026년 4월 Google DeepMind가 공개한 최신 오픈 모델 패밀리로, Gemini 3 연구 성과를 기반으로 개발되어 기술 신뢰도가 높고, Apache 2.0 라이선스로 상업적·공공 활용이 자유롭습니다. 35개 이상의 언어 기본 지원 및 140개 이상 언어 사전학습으로 한국어 처리 성능도 기대할 수 있으며, 128K 토큰의 넓은 컨텍스트 윈도우를 통해 장문의 행정 문서도 단일 프롬프트로 처리할 수 있습니다.

03학습 데이터와 튜닝 방식

Gemma 4는 Google DeepMind의 Gemini 3 연구를 기반으로 구축된 모델로, 140개 이상의 언어로 이루어진 대규모 웹 텍스트·코드·수학·과학 데이터로 사전학습되었습니다. 구체적인 사전학습 데이터셋 세부 목록은 공식 모델 카드에 완전히 공개되어 있지 않으나, 고품질 필터링을 거친 공개 웹 데이터, 코드 저장소, 다국어 문서가 포함된 것으로 알려져 있습니다.

튜닝 방식:
- Instruction Tuning (SFT): 지시 수행 능력을 위한 지도 미세조정
- RLHF / RLAIF: 인간 피드백 또는 AI 피드백을 활용한 강화학습으로 안전성 및 유용성 향상
- Thinking Mode (내장 추론): 단계별 사고(Chain-of-Thought) 능력을 활성화하는 별도 설정 지원
- 멀티모달 정렬: 비전 인코더(약 150M 파라미터)와 오디오 인코더를 언어 모델과 함께 공동 학습하여 이미지·음성 입력을 텍스트 임베딩 공간에 정렬

안전 필터링을 위한 책임 있는 AI(Responsible AI) 프로세스도 Google DeepMind의 내부 프로토콜에 따라 적용되었습니다.

04파라미터 및 구조적 특징

ArchitectureGemma 4 Dense — 하이브리드 어텐션(로컬 슬라이딩 윈도우 512토큰 + 글로벌 어텐션 교차 배치, 마지막 레이어는 항상 글로벌)
Context Length128,000 토큰 (128K)
Hidden Size공개 범위 내 확인 어려움 (Gemma 4 E4B 계열 Dense 구조; 공식 config.json 참조 권장)
Layers공개 범위 내 확인 어려움 (공식 config.json 참조 권장)
Attention Heads글로벌 레이어: 통합 Keys·Values(unified KV) 적용 + Proportional RoPE(p-RoPE) 사용으로 장문 컨텍스트 메모리 최적화
TokenizerSentencePiece 기반 BPE — Gemma 계열 공통 토크나이저, 다국어(140개 이상 언어) 지원 어휘 포함

05모델 파일 구성

Hugging Face 저장소 google/gemma-4-E4B-it의 주요 파일 구성은 다음과 같습니다.

- config.json: 모델 아키텍처 정의 파일. hidden_size, num_hidden_layers, num_attention_heads, vocab_size, max_position_embeddings(128K) 등 핵심 하이퍼파라미터 포함
- tokenizer.json / tokenizer_config.json: SentencePiece 기반 BPE 토크나이저 설정. 다국어 어휘 정보 및 특수 토큰 매핑 포함
- special_tokens_map.json: <bos>, <eos>, <pad> 등 특수 토큰 정의. Gemma 4는 Thinking Mode용 채널 토큰(<|channel>thought)도 포함
- model.safetensors / model-XXXXX-of-XXXXX.safetensors: 실제 모델 가중치. safetensors 형식은 pickle 보안 취약점 없이 빠른 로딩 지원. 대용량의 경우 샤드(shard) 분할 저장
- generation_config.json: 추론 시 기본 생성 파라미터(temperature, top_p, max_new_tokens 등) 설정
- preprocessor_config.json: 이미지·오디오 전처리기 설정 (비전 인코더 입력 해상도, 오디오 샘플링 레이트 등 멀티모달 처리 파라미터 포함)

06활용 가능 업무

Gemma 4 E4B는 다음과 같은 공공·행정 업무에 활용할 수 있습니다.

① 행정 문서 요약 및 질의응답
128K 토큰 컨텍스트를 활용해 수십 페이지 분량의 내부 지침서·법령·보고서를 입력한 뒤, 담당자가 자연어 질문으로 필요 조항을 즉시 검색·요약할 수 있습니다.

② 문서 이미지 OCR 및 내용 분석
스캔된 공문서·민원 서류·영수증 이미지를 입력하면 다국어 OCR, 표·차트 이해, 필기 인식 기능을 통해 텍스트를 추출하고 내용을 분석할 수 있습니다.

③ 음성 민원 접수 및 전사(ASR)
E4B 전용 오디오 인코더를 활용해 민원인의 음성을 텍스트로 변환(ASR)하고, 곧바로 내용 분류·답변 초안 생성까지 단일 파이프라인으로 처리 가능합니다.

④ RAG(검색 증강 생성) 기반 내부 지식 챗봇
내부 규정집·FAQ를 벡터 DB에 적재하고 E4B를 생성 엔진으로 연결하면, 직원용 행정 규정 질의응답 챗봇을 온프레미스 환경에서 구축할 수 있습니다.

⑤ 보고서·공문 초안 작성 지원
Function Calling 기능을 활용해 기안 양식 시스템과 연동하거나, 주요 내용을 입력하면 공문 형식의 초안을 자동 생성하는 에이전틱 워크플로우 구성이 가능합니다.

07한계와 주의사항

① 한국어 지원 수준
140개 이상의 언어 사전학습 및 35개 언어 기본 지원을 표방하나, 한국어 전용 벤치마크 성능은 공식적으로 충분히 검증되지 않았습니다. 행정 특화 한국어 용어, 법령 문체, 공문 형식 등에서는 별도 파인튜닝(LoRA/QLoRA) 또는 시스템 프롬프트 엔지니어링이 필요할 수 있습니다.

② 환각(Hallucination) 가능성
모든 LLM과 마찬가지로 사실과 다른 내용을 자신감 있게 생성하는 환각 현상이 발생할 수 있습니다. 법령 해석·수치 정보·날짜 등 정확도가 중요한 공공 업무에서는 반드시 RAG 또는 전문가 검토를 병행해야 합니다.

③ 메모리 요구량
"유효 4B" 모델이지만 Per-Layer Embeddings(PLE) 구조로 인해 실제 디스크 및 메모리 적재 용량은 단순 4B 모델보다 높습니다. 저사양 엣지 디바이스 배포 시 양자화(INT4/INT8) 적용이 필요합니다.

④ Thinking Mode 비활성화 동작 차이
E4B는 Thinking Mode를 비활성화해도 E2B와 함께 다른 대형 모델과 달리 빈 사고 블록을 생략하는 특수한 동작 방식을 가집니다. 멀티턴 대화 구현 시 이전 사고 내용을 히스토리에 포함하지 않도록 주의가 필요합니다.

⑤ 라이선스 제한 사항
Apache 2.0 라이선스로 상업적 활용 및 공공기관 배포가 자유롭습니다. 다만, 출력물에 AI 생성 표기 의무, Google 이용 약관 준수 등 별도 정책 확인이 권장됩니다.

08종합 의견

Gemma 4 E4B는 보안 민감도가 높은 공공기관이 온프레미스·온디바이스 환경에서 운영 가능한 멀티모달 소형 언어 모델 중 현재 시점에서 가장 균형 잡힌 선택지 중 하나입니다. Apache 2.0 라이선스, 기술 신뢰도 높은 Google DeepMind 출처, 128K 컨텍스트, 텍스트·이미지·오디오 통합 처리 능력은 공공 행정 디지털화 과제에 직접적으로 대응합니다.

도입 적합 조건:
- 개인정보·기밀문서를 외부 클라우드로 전송할 수 없는 온프레미스 환경
- 스캔 문서 디지털화, 음성 민원 접수, 내부 지식 Q&A 등 멀티모달 업무 요구
- 소수의 GPU 또는 고성능 CPU 서버로 운영해야 하는 소규모 기관

추가 검증 및 POC 권장 사항:
① 한국어 행정 문서(공문, 법령, 보도자료)에 대한 요약·질의응답 정확도 벤치마크 수행
② 기관 내부 도메인 데이터를 활용한 LoRA 파인튜닝 효과 검증
③ 보안 요구사항에 맞는 온프레미스 추론 서버(vLLM, llama.cpp) 성능 및 처리량 테스트
④ 환각 감소를 위한 RAG 파이프라인 연계 및 출력 검증 프로세스 설계

단계적 도입(소규모 부서 파일럿 → 성능 검증 → 기관 전체 확대)을 권장하며, 초기 POC 단계에서는 Ollama 또는 LM Studio를 활용한 데스크톱 수준 테스트로 충분히 검증 가능합니다.
