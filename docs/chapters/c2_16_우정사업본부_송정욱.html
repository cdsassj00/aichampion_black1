01모델 선택 이유

공공기관 업무 자동화에서 언어모델 도입의 최대 장벽은 보안과 비용이다. 클라우드 기반 대형 모델(GPT-4, Claude 등)은 민감 문서의 외부 전송 문제로 망분리 환경에 적용할 수 없고, 자체 GPU 클러스터 운영은 수억 원의 초기 투자를 수반한다. EXAONE 3.5 2.4B-Instruct-GGUF(Q6_K)를 선택한 이유는 다음 세 가지 기준을 동시에 충족하는 유일한 한국어 특화 SLM이기 때문이다. 첫째, 한국어 형태소 최적화 토크나이저다. SuperBPE 기반 102,400개의 대규모 어휘 사전은 교착어 특성상 단어 분절이 복잡한 한국어를 처리할 때, GPT-2 계열 50,000 vocab 대비 토큰 소비량을 약 30% 절감한다. 행정 문서는 단문보다 복합 명사·조사 결합 구조가 빈번하므로, 이 효율성 차이는 32K 컨텍스트 창 활용도와 추론 속도 모두에 직접적 영향을 미친다. 둘째, 폐쇄망 단독 실행 가능성이다. Q6_K 양자화 적용 시 모델 가중치가 약 2.19GB로 압축되어, 인터넷 연결 없이 16GB RAM 데스크탑에서 Ollama + llama.cpp 스택만으로 구동된다. 이는 우정사업본부 내부망 환경에서 외부 API 의존 없이 민감 데이터를 처리할 수 있음을 의미한다. 셋째, 라이선스 적합성이다. EXAONE AI Model License Agreement 1.1-NC는 비영리 연구 및 내부 행정 활용을 허용하므로, 공공기관의 내부 업무 자동화 목적에 법적 문제가 없다.

03학습 데이터와 튜닝 방식

사전학습(Pre-training): 한국어와 영어를 5:5 비율로 혼합한 대규모 멀티링궐 코퍼스로 1차 사전학습을 수행했다. 이후 텍스트에서 핵심 개념을 추출한 Taxonomy(지식 계층) 기반의 고품질 데이터를 활용해 2차 정밀 학습(Continual Pre-training)을 진행함으로써, 일반 도메인 코퍼스만 사용했을 때의 사실 오류율과 논리 일관성 문제를 보완했다. 이 2단계 사전학습 구조는 Phi-3 시리즈의 "textbook-quality data" 전략과 유사한 접근법으로, 파라미터 효율 대비 추론 품질을 극대화하는 데 효과적이다. SFT(Supervised Fine-Tuning): 약 800만 건의 웹 코퍼스에서 증류된 핵심 지식 쌍(instruction–response)을 학습 데이터로 활용했다. 단순 Q&A 형태를 넘어, 다단계 사고 과정(chain-of-thought 형식)이 포함된 데이터를 통해 복합 지시문 이행 능력을 강화했다. 이 과정은 모델의 Format Following 능력, 즉 JSON 구조화 출력이나 마크다운 형식 준수 등의 실무 활용에 직결되는 역량을 결정한다. DPO(Direct Preference Optimization): RLHF의 보상 모델 학습 비용과 학습 불안정성을 제거한 DPO를 사용해 인간 선호도를 정렬했다. 선호(chosen)·비선호(rejected) 쌍으로 구성된 비교 데이터를 통해 유해 출력 억제, 정직성 향상, 한국어 경어체 준수 등의 행동 정렬을 수행했다. SFT가 "무엇을 말할 것인가"를 학습시킨다면 DPO는 "어떻게 말할 것인가"의 품질과 안전성을 정제하는 단계다. 양자화(Quantization): GGUF Q6_K 방식은 6비트 양자화로, 가중치를 블록 단위로 분할한 뒤 각 블록에서 스케일 팩터(k-quants)를 추출해 정밀도 손실을 최소화한다. BF16 원본 대비 메모리 점유는 약 63% 수준(2.19GB)으로 줄이면서, perplexity 증가폭은 Q4_K_M 대비 절반 이하로 유지한다. CPU 추론 엔진인 llama.cpp에서 AVX2 명령어 셋을 활용해 행렬 연산을 벡터화하므로, 외장 GPU 없이도 실용적인 추론 속도를 달성한다.

04파라미터 및 구조적 특징

ArchitectureDecoder-only Transformer (EXAONE 자체 구조 · LLaMA 계열과 유사한 RoPE 위치 인코딩 적용)
Context Length32,768 토큰 (32K) — 수십 페이지 분량의 행정 보고서 단일 패스 처리 가능
Hidden Size2,048 (config.json: hidden_size: 2048)
Layers30 Transformer Decoder Blocks — 2.4B 규모 대비 깊은 레이어로 추론 품질 확보
Attention HeadsQ-heads: 32 / KV-heads: 8 → GQA(Grouped Query Attention) 적용. KV-cache 메모리를 75% 절감하여 장문 처리 시 메모리 효율 극대화
TokenizerSuperBPE 기반 · Vocab size: 102,400 — 한국어 형태소 분절 최적화로 타 글로벌 모델 대비 한국어 토큰 효율 약 30% 향상

05모델 파일 구성

config.json — 모델의 정적 구조를 정의하는 청사진 파일이다. hidden_size: 2048, num_hidden_layers: 30, num_attention_heads: 32, num_key_value_heads: 8 등 아키텍처 하이퍼파라미터가 모두 기록되며, 추론 엔진이 가중치 텐서를 올바르게 로드하기 위한 참조 값으로 사용된다. tokenizer.json + tokenizer_config.json — 102,400개 어휘 사전과 SuperBPE 병합 규칙, 특수 토큰([BOS], [EOS], [PAD]) 매핑 정보를 담는다. 한국어 자모 분해 패턴과 교착어 처리를 위한 서브워드 규칙이 포함되어 있어, 이 파일이 곧 EXAONE의 한국어 효율성의 핵심 근거다. generation_config.json — 디코딩 전략의 기본값을 정의한다. temperature: 1.0, repetition_penalty: 1.0 등 권장 샘플링 파라미터가 명시되어 있다. 행정 문서 생성처럼 결정론적 출력이 필요한 경우 temperature: 0.1~0.3, do_sample: false로 변경해 greedy decoding에 가깝게 운용하는 것이 바람직하다. model.gguf (Q6_K) — 6비트 k-quants 방식으로 압축된 실제 가중치 파일이다. GGUF(GPT-Generated Unified Format)는 단일 파일에 메타데이터·토크나이저·가중치를 모두 통합한 포맷으로, llama.cpp 및 Ollama가 직접 로드한다. 파일 크기 약 2.19GB이며 별도 변환 없이 CPU 추론 환경에서 즉시 사용 가능하다. special_tokens_map.json — 시스템 프롬프트 구분자([|system|], [|assistant|] 등 EXAONE 전용 채팅 템플릿 토큰)를 정의한다. Ollama 사용 시 이 템플릿이 자동 적용되나, 직접 llama.cpp를 호출할 때는 --chat-template 옵션을 명시해야 지시문 추종 성능이 보장된다.

06활용 가능 업무

① 행정 문서 초안 자동 생성: SFT 단계에서 공공 문체와 격식 구조를 학습한 EXAONE은 기안문·협조 공문·보도자료 등의 초안을 생성하는 데 즉시 활용 가능하다. 사용자가 핵심 키워드와 맥락만 제공하면 모델이 문단 구조·경어체·근거 논리를 갖춘 초안을 반환하며, 담당자는 최종 검토와 서명에만 집중할 수 있다. ② 한글(HWP/HWPX) 문서 분석 및 요약: python-hwpx 라이브러리로 본문 텍스트를 추출한 뒤 EXAONE의 32K 컨텍스트 창에 투입해 대형 보고서 전문을 단일 패스로 요약할 수 있다. 32K 컨텍스트는 A4 약 40~50페이지 분량에 해당하며, 연간 보고서·감사 결과서 수준의 문서를 통째로 처리하기에 충분하다. ③ 한셀(Hancel) 데이터 구조화 처리: InputBox API를 통해 스프레드시트의 원시 데이터를 모델에 전달하면, 통계 해석·이상값 탐지·테이블 정규화·요약문 작성 등의 분석 업무를 자동화할 수 있다. 특히 정형화되지 않은 집계 데이터를 자연어 보고서 형태로 변환하는 시나리오에서 효과적이다. ④ 내부 Q&A / RAG 기반 정책 검색 시스템: 내부 지침서·법령 해설서를 벡터 DB에 임베딩하고, EXAONE을 RAG(Retrieval-Augmented Generation) 파이프라인의 리더 모델로 배치하면, 직원이 자연어로 질의했을 때 근거 문서를 인용하며 답변하는 폐쇄형 지식 검색 시스템을 구현할 수 있다. 외부 통신이 없으므로 기밀 정책 문서도 안전하게 처리된다. ⑤ 폐쇄망 AI 비서 (Ollama 단독 구동): 인터넷이 완전 차단된 망분리 환경에서도 Ollama 프레임워크를 통해 모델을 로컬 API 서버로 구동할 수 있다. i5-14500 CPU 환경에서 초당 약 20~30 토큰의 생성 속도를 달성하여, 질의 응답·초안 작성·번역 등 비서 업무의 실시간 응답성을 확보한다.

07한계와 주의사항

① 환각(Hallucination) 위험: 2.4B 규모의 SLM은 대형 모델 대비 사실 저장 용량(factual capacity)이 제한적이어서, 학습 데이터에 포함되지 않은 최신 법령 개정·인사 정보·사건 세부 내용에 대해 그럴듯하지만 부정확한 정보를 생성할 수 있다. RAG 파이프라인을 통해 검색된 문서를 컨텍스트로 제공하거나, 모델 출력에 반드시 출처 기반 검증 단계를 추가해야 한다. 자율적 사실 판단이 필요한 민원 결정·법적 해석 업무에는 단독 적용을 지양한다. ② 모델 용량 대비 복합 추론 한계: 수학적 다단계 연산(multi-step arithmetic)이나 고도의 논리 체인이 필요한 작업에서는 7B 이상 모델 대비 오류율이 유의미하게 높아진다. KoMT-Bench 7.24점은 7B급에 근접한 수준이나 절대적 정밀도가 요구되는 예산 계산·통계 분석에는 출력 값 검증이 필수다. ③ Q6_K 양자화 정밀도 손실: BF16 원본 대비 가중치 정밀도가 낮아져 perplexity가 소폭 상승한다. 일반 문서 생성 업무에서는 체감 차이가 미미하나, 코드 생성이나 JSON 스키마 준수 등 형식 정확도가 중요한 작업에서는 Q8_0 또는 BF16 추론 환경(GPU 필요)과의 품질 비교 검토가 권장된다. ④ 라이선스 제약: EXAONE AI Model License 1.1-NC는 외부 사용자 대상 상업적 서비스(민원 챗봇 공개 서비스 등)에 적용할 경우 별도 협의가 필요하다. 내부 직원 전용 도구로만 운용하는 한 문제가 없으나, 서비스 범위 확장 시 LG AI Research와의 라이선스 협의를 사전 검토해야 한다. ⑤ 편향 및 공정성: 한국어·영어 5:5 혼합 학습 특성상 특정 지역·직군에 대한 미세한 서술 편향이 존재할 수 있다. 대민 서비스 문서를 모델이 직접 생성할 경우, 공정성 감사(bias audit) 프로세스를 별도 수립하는 것이 바람직하다.

08종합 의견

EXAONE 3.5 2.4B-Instruct-GGUF(Q6_K)는 현재 공개된 한국어 특화 SLM 중 온프레미스 폐쇄망 환경에서의 실용성 대비 성능비가 가장 뛰어난 모델이라고 평가한다. 핵심 근거는 세 가지다. 첫째, LogicKor 8.51점·KoMT-Bench 7.24점이라는 벤치마크 수치는 2.4B 규모가 7B급 성능 영역에 진입했음을 보여준다. 이는 SuperBPE 기반 고효율 토크나이저와 2단계 사전학습(Taxonomy-driven Continual PT)이 파라미터 효율을 극대화한 결과다. 둘째, GQA 적용으로 KV-cache 메모리 사용량을 MHA 대비 75% 절감해 32K 장문 컨텍스트를 16GB RAM 환경에서 현실적으로 운용할 수 있다. 셋째, GGUF Q6_K 포맷은 BF16 대비 perplexity 손실을 최소화하면서 설치와 구동을 단일 명령어(ollama run exaone3.5:2.4b)로 완결한다는 점에서 IT 자원이 제한된 공공기관 환경에 적합하다. 도입 권고 조건: 단순 반복 문서 초안 작성·내부 Q&A·보고서 요약 등 "초안 생성 후 인간 검토" 구조의 업무에 먼저 적용하고, 모델 출력을 최종 의사결정에 바로 반영하지 않는 Human-in-the-loop 원칙을 준수해야 한다. 향후 과제: python-hwpx와 Ollama REST API를 연동한 한글 문서 분석 파이프라인 구축, 도메인 특화 SFT(우정 업무 관련 지침서·민원 응대 데이터)를 통한 LoRA 파인튜닝 검토, 그리고 RAG 파이프라인 통합 시 청크 전략(청크 크기·오버랩)과 리트리버 성능 최적화가 다음 단계 과제로 권고된다.
