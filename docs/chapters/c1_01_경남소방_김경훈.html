01

모델 선택 이유

소방 조직에서 SLM을 도입하려면 일반 행정과는 다른 세 가지 조건을 동시에 만족해야 한다. 현장 무선·119 신고 데이터의 외부 유출 차단, 한국어 구어체와 소방 전문용어의 동시 이해, 그리고 소방서·119안전센터급 하드웨어에서 구동 가능한 경량 규모이다. Kanana-1.5-2.1B-Instruct-2505는 이 세 조건을 모두 충족한다.

첫째, Apache 2.0 라이선스로 공개되어 상업적·공공기관 활용에 제약이 없다. LG의 EXAONE이나 카카오의 1세대 Kanana-Nano가 CC-BY-NC-4.0인 것과 달리, Kanana 1.5 시리즈 중 2.1B·8B는 Apache 2.0이라 소방 정보시스템과의 통합·재배포에 법적 부담이 적다.

둘째, 한국어 평가에서 의미 있는 성능을 보인다. KoMT-Bench 6.54점, KMMLU(CoT) 32.93점, HAERAE 77.46점으로 동급(2B) 모델 중에서는 한국어 행정·법률·교양 도메인 이해도가 높은 편이며, 영어(MMLU 53.87, MT-Bench 7.01)도 동반 가능해 외국인 신고·국제 협력 문서 등 이중언어 처리에도 적합하다.

셋째, 2.1B 파라미터로 BF16 기준 약 4GB의 VRAM 만으로 추론 가능하며, Q4 양자화 시 1~2GB 수준으로 소방서·119안전센터 단위의 일반 GPU·사내 서버에서 직접 구동할 수 있어 119 신고 내용·환자 정보·현장 보고서 등 민감 데이터를 외부 API에 보내지 않고도 분류·요약·검색을 구축할 수 있다. 나아가 네이티브 32K, YaRN 적용 시 128K 컨텍스트를 지원해 소방기본법·화재예방법·시행규칙·매뉴얼 같은 장문 법령을 한 번에 입력해 질의응답이 가능하다는 점이 결정적이었다.

03

학습 데이터와 튜닝 방식

Kanana는 한국어·영어 이중언어 모델로, Technical Report(arXiv 2502.18934)에 따르면 사전학습 단계에서 고품질 데이터 필터링(high-quality data filtering), 단계적 사전학습(staged pre-training), depth up-scaling, pruning & distillation을 결합한 '컴퓨트 효율적' 학습 전략을 사용했다. 구체적인 데이터셋 명세(웹·도서·코드 비율, 출처 등)는 공개 범위 내에서는 일부만 확인된다.

후처리 단계에서는 일반적인 채팅 모델의 표준 파이프라인을 따른다.

Supervised Fine-Tuning 지시-응답 쌍을 활용한 SFT로 명령 수행 능력 학습
Preference Optimization 인간 선호도 기반 정렬(선호도 최적화). 1.5에서 후처리 프로세스 정제
1.5 버전 강화 영역 코딩 · 수학 · Function Calling · 자연스러운 대화 품질
RLHF / DPO 명시 여부 모델 카드 공개 범위에서는 명시되지 않음 (선호도 최적화 표현으로 통칭)
개인정보 정책 카카오 사용자 데이터는 사전학습·후처리 어디에도 포함되지 않음

특히 모델 카드에 명시된 "Neither the pre-training nor the post-training data includes Kakao user data" 문구는 공공기관 도입 시 개인정보 보호 리스크 검토 측면에서 의미 있는 진술이다. 다만 한국 행정 도메인(법령·고시·민원)에 특화된 데이터로 학습되었는지는 공개되지 않았으므로, 실제 도입 전 도메인 파인튜닝(LoRA·QLoRA)이 권장된다.

05

모델 파일 구성

Hugging Face 저장소에 업로드된 주요 파일과 각 파일의 역할은 다음과 같다. 이 파일들은 모두 AutoModelForCausalLM.from_pretrained() 호출 시 자동으로 읽혀 모델을 복원한다.

config.json
모델 아키텍처 하이퍼파라미터 정의. architectures, hidden_size, num_hidden_layers, num_attention_heads, num_key_value_heads, max_position_embeddings, rope_theta, vocab_size 등. 추론 엔진(transformers, vLLM, SGLang)이 모델 구조를 인식하는 단일 진입점.
model.safetensors
모델 가중치(weights)를 안전한 텐서 포맷으로 저장한 파일. BF16 기준 약 4.2GB. pickle 기반 .bin과 달리 임의 코드 실행 위험이 없어 보안 민감 환경(공공·금융)에서 권장되는 포맷.
tokenizer.json
BPE 토크나이저의 vocab과 merge 규칙, pre/post-tokenizer 정의. AutoTokenizer.from_pretrained()가 이 파일만 있어도 동작하도록 만들어진 fast tokenizer 포맷.
tokenizer_config.json
토크나이저 메타데이터와 함께 chat_template(Jinja2)을 담는다. apply_chat_template()이 system/user/assistant role을 모델 입력 포맷으로 변환할 때 참조하는 핵심 파일.
special_tokens_map.json
BOS, EOS, PAD, UNK 등 특수 토큰의 ID와 문자열 매핑. 대화 종료(EOS)·패딩 토큰 충돌 방지를 위한 명세.
generation_config.json
기본 생성 파라미터(temperature, top_p, top_k, repetition_penalty, max_new_tokens 등). model.generate() 호출 시 인자를 지정하지 않으면 이 파일의 값이 사용된다.
README.md (Model Card)
모델 카드 본문. 용도·한계·벤치마크·라이선스·인용 정보 포함. Hugging Face Hub 페이지에서 렌더링되는 표시면이자 도입 의사결정의 1차 근거 문서.
06

활용 가능 업무

2.1B 규모는 단독으로 ChatGPT급 추론을 대체하기 어렵지만, 정형화된 소방 행정 업무 자동화와 소방 RAG 백엔드 보조 모델로는 충분한 성능을 낸다. 다음은 경남소방본부에서 즉시 PoC 가능한 시나리오다.

USECASE / 01

119 신고 텍스트 분류

STT로 변환된 119 신고 텍스트를 화재·구조·구급·기타로 자동 분류하고 긴급도(코드 적색·황색·녹색) 1차 판정 보조. 2.1B로도 분류 태스크는 충분히 견딘다.

USECASE / 02

출동 보고서 초안 생성

현장 대원이 입력한 단편 메모와 무전 기록을 바탕으로 화재·구조·구급 출동 보고서 초안을 자동 작성. 대원은 사실관계 확인·수정만 담당해 행정 시간을 단축.

USECASE / 03

소방법령·예방규정 RAG

소방기본법·화재예방법·위험물안전관리법·건축법 시행규칙을 임베딩 인덱싱한 뒤 Kanana를 생성기로 사용. 예방·검사·민원 담당자가 조항 단위로 질의응답. 온프레미스 배포로 외부 유출 위험 차단.

USECASE / 04

화재 조사 보고서 요약

장문(수십~수백 페이지)의 화재 조사·감식 보고서를 32K 컨텍스트에 통째로 입력해 발화 원인·연소 확산·인명피해·재발 방지 권고를 요약 추출. 128K YaRN 적용 시 연도별 통합 보고서 분량까지 처리 가능.

USECASE / 05

안전 점검 체크리스트 자동화

다중이용업소·위험물 시설 현장 점검표를 텍스트로 입력하면 지적사항을 항목별로 정리하고 행정처분 절차 초안을 생성. 반복 작성 업무 시간을 줄여 현장 점검 빈도 확보.

USECASE / 06

외국인 신고·민원 1차 대응

영문 신고·민원 텍스트를 한국어로 1차 번역해 상황실에 전달, 한국어 응대 초안을 영문으로 변환해 회신 보조. 전문 통역의 대체는 아니나 골든타임 확보용 시안으로 유용.

Kanana-1.5-2.1B는 '완성형 AI 비서'가 아니라 '한국어 소방 도메인 위에 얹는 가벼운 추론 엔진'이다. RAG·룰 기반 검증·현장 대원 휴먼인더루프와 결합할 때 비로소 가치가 살아난다.
07

한계와 주의사항

  • 규모 한계. 2.1B 파라미터는 복잡한 다단계 추론·전문 법률 해석에서 7B~70B 모델 대비 명백히 약하다. 단독으로 정책 결정을 대신할 수 있는 수준은 아니다.
  • 환각(Hallucination). 존재하지 않는 법조문·통계·인명을 그럴듯하게 만들어낼 수 있다. 행정 문서에 출력을 그대로 사용하지 말고 RAG·검증 룰·담당자 리뷰를 반드시 결합해야 한다.
  • 한국어 도메인 깊이. 일반 한국어는 우수하나, 화재 조사·위험물·구급 의료·소방시설 같은 소방 전문 도메인 용어는 LoRA·QLoRA 기반 추가 파인튜닝이 권장된다.
  • 학습 데이터 투명성. 공개된 사전학습 데이터 출처 목록이 제한적이라 저작권·개인정보 유래에 대한 자체 위험 평가가 필요하다. (단, 카카오 사용자 데이터 미포함은 명시됨)
  • 평가 데이터 한계. KMMLU, KoMT-Bench 같은 공개 벤치마크 성능이 곧 '현장 행정 업무 성능'은 아니다. 반드시 자체 평가셋으로 별도 검증해야 한다.
  • 장문 컨텍스트 비용. 128K YaRN 사용 시 KV-Cache 메모리가 급증한다. 평균 입력 길이를 측정해 32K 네이티브로 충분한지 먼저 검토할 것.
  • 상업 활용. Apache 2.0이라 상업·공공 활용 모두 가능하지만, 파생 모델 배포 시 NOTICE·LICENSE 파일 동봉 의무를 준수해야 한다.
  • 보안. 외부 API가 아닌 온프레미스 배포가 가능한 것이 강점이나, 모델 가중치 자체의 무결성 검증(safetensors 해시 확인)과 프롬프트 인젝션 대응 가드레일을 별도로 구축해야 한다.
08

종합 의견

Kanana-1.5-2.1B-Instruct-2505는 "한국어로 동작하는 가장 가벼운, 그러나 진지하게 쓸 만한 SLM"이라는 좌표 위에 있다. 경남소방본부의 입장에서 이 모델은 '단독으로 도입할 모델'이 아니라 기존 119 신고 접수·출동·예방 업무 파이프라인 안에 끼워 넣어 행정 부담을 낮추는 보조 엔진으로 검토하는 것이 합리적이다.

도입이 적합한 조건은 다음과 같다.

  • 온프레미스 강제. 119 신고 내용·환자 정보·현장 무전이 포함되어 외부 API 호출이 금지된 환경.
  • 한국어 비중 95% 이상. 영어 의존도가 낮은 소방 현장 텍스트에 최적.
  • RAG·룰 기반 검증과 결합 가능. 모델 단독 출력에 의존하지 않는 아키텍처 (소방법령 인덱싱 필수).
  • 분류·요약·초안 작성 중심. 화재 원인 판정·인명피해 추정 같은 책임 추론은 상위 모델·전문가에 위임.

도입 전 필수 PoC 항목: ① 자체 소방 한국어 평가셋 구축(119 신고 100건·출동 보고서 100건·예방 민원 100건)으로 KoMT-Bench 외 실측 성능 검증, ② 소방 도메인 LoRA 파인튜닝 효과 측정(베이스 vs LoRA 적용 후 분류·요약 정확도 차이), ③ 32K 입력 시 추론 지연·메모리 측정(소방서 단위 서버 사양 검증), ④ 프롬프트 인젝션·개인정보(환자명·주소·주민번호) 노출 위험 레드팀 테스트, ⑤ 감사·법무 부서와 Apache 2.0 라이선스 적용 범위 사전 협의, ⑥ 119 신고 데이터의 학습·로그 보관 정책에 대한 정보공개·개인정보보호위원회 사전 검토.

결론적으로, 외산 폐쇄 API에 대한 정책적·예산적 의존도를 낮추면서 소방 행정 디지털 전환을 시작하려는 광역소방본부라면 Kanana-1.5-2.1B는 국산 SLM PoC의 출발선으로 가장 합리적인 선택지 중 하나다. 다만 '한국에서 만든 모델'이라는 점이 곧 '소방 도메인에 최적화된 모델'을 의미하지 않으므로, 도메인 검증과 LoRA 추가 학습은 협상의 여지 없이 필요한 단계로 간주해야 한다.
