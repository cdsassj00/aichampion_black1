01모델 선택 이유

공공기관에는 민원인이 제출한 스캔 PDF, 증빙서류 이미지, 과거 종이문서 등 비정형 문서가 다수 존재하며, 이들 문서는 구조화된 데이터베이스로 즉시 활용되기 어렵다. DeepSeek-OCR-2는 단순 문자 인식에 그치지 않고 문서의 읽기 순서, 표, 수식, 복잡한 레이아웃을 함께 해석하도록 설계된 OCR 특화 비전-언어 모델이므로, 비정형 행정문서를 검색·분석 가능한 텍스트 데이터로 전환하는 데 적합하다고 판단하여 선택하였다.

03학습 데이터와 튜닝 방식

DeepSeek-OCR-2는 DeepSeek-OCR와 동일한 데이터 소스를 기반으로 학습되었으며, OCR 1.0, OCR 2.0, 일반 비전 데이터를 혼합해 사용하였다. 전체 학습 혼합물 중 OCR 데이터 비중은 80%이고, OCR 1.0 데이터는 텍스트·수식·표 페이지를 3:1:1 비율로 샘플링하도록 조정하였다. 학습은 ① DeepEncoder V2 사전학습, ② Query Enhancement, ③ Decoder Specialization의 3단계 파이프라인으로 수행되었다. 1단계에서는 약 1억 개의 이미지-텍스트 쌍을 이용해 시각 특징 추출, 토큰 압축, 토큰 재정렬 능력을 학습했고, 2단계에서는 시각 토크나이저를 고정한 채 LLM형 인코더와 디코더를 공동 최적화했으며, 3단계에서는 인코더를 고정하고 디코더만 추가 학습하였다. 논문 공개 범위에서는 RLHF, DPO, LoRA, QLoRA 적용 여부는 확인되지 않는다.

04파라미터 및 구조적 특징

ArchitectureDeepEncoder V2 + DeepSeek-3B MoE Decoder. Vision Tokenizer는 SAM-base(80M)+Conv, Vision Encoder는 Qwen2-0.5B 기반 LLM형 구조
Context Length8192 토큰(max_position_embeddings)
Hidden Size1280(language decoder), 896(vision encoder 출력 차원)
Layers12(language decoder); SAM ViT-B도 12 layers
Attention Heads10 attention heads / 10 key-value heads(language decoder)
Tokenizertokenizer.json 기반 토크나이저, vocab size 129,280(세부 알고리즘은 공개 파일상 별도 명시 없음)

05모델 파일 구성

Hugging Face 저장소의 주요 파일은 다음과 같다. config.json은 전체 모델 구조, 언어 디코더 설정, 비전 인코더 설정, 토큰 수, hidden size, layer 수 등을 정의한다. model-00001-of-000001.safetensors는 실제 학습 가중치 파일이며, model.safetensors.index.json은 가중치 인덱스 정보를 담는다. tokenizer.json과 tokenizer_config.json은 텍스트 토크나이저와 토큰화 설정을, special_tokens_map.json은 BOS/EOS 등 특수 토큰 매핑을 정의한다. processor_config.json은 이미지 입력 전처리 설정을 관리한다. 또한 deepencoderv2.py, modeling_deepseekocr2.py, modeling_deepseekv2.py, configuration_deepseek_v2.py 등은 커스텀 모델 구조와 추론 동작을 구현하는 코드 파일이다.

06활용 가능 업무

공공기관에서는 ① 민원 첨부서류·스캔 PDF·증빙 이미지의 자동 텍스트화, ② 표·수식·복합 레이아웃이 포함된 행정문서의 구조적 추출, ③ 기존 종이 기록물의 디지털 전환, ④ OCR 결과를 기반으로 한 RAG 검색 데이터 구축, ⑤ 문서 분류·요약·질의응답을 위한 전처리 단계에 활용할 수 있다. 예를 들어 병무 행정에서는 외국어 민원서류, 병역 관련 증빙자료, 판정표, 각종 신청서 이미지를 텍스트와 구조화 데이터로 변환한 뒤, 후속 LLM 또는 검색 시스템과 연계해 담당자 검토 시간을 줄이는 보조 도구로 사용할 수 있다.

07한계와 주의사항

첫째, 주된 평가와 학습 설명이 문서 OCR 중심이므로 일반 시각 추론이나 한국어 행정문서에서의 성능은 별도 검증이 필요하다. 둘째, 논문에서도 신문처럼 텍스트가 매우 조밀한 문서에서는 성능 저하 여지가 있음을 언급하였고, 한국어·혼합 언어·저화질 스캔·도장·필기체 문서에서도 기관 자체 데이터로 평가해야 한다. 셋째, OCR 결과가 항상 정답은 아니므로 법적 효력이 있는 행정처리에는 사람의 검토 절차를 유지해야 한다. 넷째, 개인정보가 포함된 민원 문서를 처리할 경우 외부 전송 금지, 폐쇄망 배포, 접근권한 통제, 로그 관리 등 보안 요건을 우선 검토해야 한다. 다섯째, Apache-2.0 라이선스이지만 커스텀 코드(trust_remote_code=True)를 사용하므로 운영 도입 전 코드 보안 점검과 의존성 검토가 필요하다.

08종합 의견

DeepSeek-OCR-2는 일반 챗봇용 SLM이라기보다 비정형 문서를 텍스트 자산으로 전환하는 전처리 모델로서 가치가 크다. 특히 문서의 읽기 순서를 재구성하는 DeepEncoder V2를 적용해 기존 OCR보다 복잡한 레이아웃 처리에 강점을 보인다는 점에서, 공공기관의 문서 디지털화·검색 고도화·RAG 구축에 유용할 수 있다. 다만 실제 도입 전에는 병무청 보유 문서 샘플을 대상으로 한 POC가 필요하며, 한국어 OCR 정확도, 표 추출 품질, 스캔 품질 변화에 따른 성능, 처리 속도, GPU 자원 요구량, 개인정보보호 및 폐쇄망 배포 가능성을 함께 검증해야 한다. 검증 결과가 양호하다면, 단독 의사결정 모델이 아니라 문서 전처리 및 담당자 업무 지원 모델로 도입하는 것이 적절하다.
