01모델 선택 이유

광진구청 행정 업무 관점에서 SLM 도입을 검토할 때 ① 클라우드 외부 LLM API 호출 시 발생하는 민원 정보 외부 유출 우려, ② 자치구 단위의 GPU 예산·인프라 한계 안에서 운영 가능한 규모, ③ 한국어 응대 정확도 — 이 세 가지 요건을 동시에 만족해야 한다. Qwen2.5-7B-Instruct는 (1) 7B급으로 단일 24GB GPU(RTX 4090) 또는 4-bit 양자화 시 12~16GB VRAM 환경에서 추론 가능하여 자치구 자체 인프라 운영이 현실적이고, (2) Apache-2.0 라이선스로 상업·내부 운영·재배포·파인튜닝 모두 제약 없이 활용 가능하며, (3) 사전학습 단계에서 한국어를 포함한 29개 다국어를 명시적으로 다루어 동급 7B 오픈모델 중 한국어 응답 품질이 안정적인 편이다. RAG 기반 조례·민원 응답 시스템 또는 내부 문서 분류·요약 보조 도구의 백본으로 평가하기 위해 본 모델을 선정했다.

03학습 데이터와 튜닝 방식

사전학습은 Qwen2의 7T → 18조(18T) 토큰으로 확장된 다국어 코퍼스를 사용한 것으로 공식 블로그(qwenlm.github.io/blog/qwen2.5)에 보고되어 있으며, 코드·수학·다국어 데이터 비중을 직전 세대 대비 증대했다. 다만 세부 데이터 출처 목록과 한국어 비중은 모델 카드·기술 문서에서 비공개이며 — 공개 범위 내에서는 확인이 어렵다. 후학습(post-training)은 ① 100만+ 인스트럭션 샘플의 대규모 SFT(Supervised Fine-Tuning)와 ② 다단계 강화학습(공식 표현은 multi-stage RL)으로 선호도 정렬을 적용했다. RLHF/DPO 정확한 비중은 공식 명기되지 않았으나 DPO 계열 오프라인 선호도 학습이 포함된 것으로 알려져 있다. LoRA·QLoRA 등 PEFT는 공개 모델 자체 학습 파이프라인이 아니라 사용자 측 도메인 적응 단계에서 활용 가능한 후속 옵션이다.

04파라미터 및 구조적 특징

ArchitectureQwen2 (Transformer 디코더 + RoPE + SwiGLU + RMSNorm + GQA + Attention QKV bias)
Context Length32,768 토큰 (네이티브) · 최대 131,072 토큰 (YaRN rope_scaling 활성)
Hidden Size3,584 (intermediate_size 18,944)
Layers28
Attention Heads28 query heads / 4 KV heads — GQA 7:1 (KV 캐시 메모리 1/7 절감)
TokenizerQwen2 BPE (tiktoken 기반) · vocab 152,064 · tie_word_embeddings=false · bfloat16

05모델 파일 구성

① config.json — 아키텍처 하이퍼파라미터(레이어·헤드·hidden·rope_theta·max_position) 정의. transformers AutoModel이 이 파일을 읽어 빈 모델 인스턴스를 만들고 가중치를 주입한다. ② tokenizer.json · tokenizer_config.json · vocab.json · merges.txt — Qwen2 BPE 토크나이저 본체. 152K vocab과 ChatML 스타일 채팅 템플릿(`apply_chat_template`)이 tokenizer_config.json의 `chat_template` 필드에 Jinja2로 정의되어 있다. ③ model-00001-of-00004.safetensors ~ model-00004-of-00004.safetensors + model.safetensors.index.json — 7.61B 가중치를 4개 shard로 분할 저장(bfloat16, 합 약 15GB). safetensors는 임의 코드 실행 위험이 없는 안전한 직렬화 포맷이라 공공기관 배포에서도 신뢰성 높음. ④ generation_config.json — 기본 디코딩 파라미터(temperature, top_p, repetition_penalty, eos_token_id 등). ⑤ special_tokens_map.json — `<|im_start|>`, `<|im_end|>` 등 ChatML 특수 토큰 매핑. ⑥ LICENSE — Apache-2.0 원문.

06활용 가능 업무

① 민원 1차 응대 챗봇 — 자치구 홈페이지·카카오톡 채널 연동. 자주 묻는 질문(전입신고·여권·인감·청소·소음·주정차) 우선 응대로 콜센터 통화량 분산. 광진구 조례·자주 묻는 질문 DB 인덱싱 후 RAG 기반 응답.

② 공문서 자동 분류·요약 — 부서 간 이관·접수 공문에 대해 분류(도시계획/복지/세무/총무/문화체육) 라벨링 + 3~5문장 요약 생성. 야간 접수분 다음 날 아침 부서장용 요약 리포트.

③ 조례·법령 RAG 질의응답 — 광진구 자치법규 + 상위법(서울특별시·국가법령정보센터) 인덱싱 후 "○○ 시 절차" 형식 질의에 근거 조항·페이지 인용 답변. 공무원 답변 부정확성 방지를 위해 출처 표시 필수화.

④ 보고서 초안 생성 — 회의록·통계 자료를 입력으로 부서장 보고용 1페이지 요약 초안. 최종 검토는 담당자가 수행하며 모델 출력은 수정 가능한 워드 파일로 export.

⑤ 다국어 민원 통역 보조 — 외국인 주민 비중이 있는 자치구 특성을 반영해 한↔영/중/일 1차 번역. 단 법적 효력 문서(증명서·고지서)는 인간 번역 필수.

⑥ 회의록 자동 작성 — STT 결과 텍스트를 정리·발언자별 요약·액션 아이템 추출. 부서 회의·구의회 속기 보조.

07한계와 주의사항

환각(Hallucination): 7B 규모 한계로 사실 검증이 필요한 행정 정보(조례 조항 번호·금액·기한)는 RAG + 인간 검수 필수. 모델 단독 응답을 그대로 민원에게 전달하는 것은 위험.

한국어 품질 격차: 다국어 학습이라 한국어 단독 학습 모델(EXAONE·Kanana 등) 대비 격식·존댓말·고유명사 처리에서 미세 오류 가능. 도입 전 KMMLU·HAE-RAE·HRM8K 등 한국어 벤치마크로 자체 검증 권장.

학습 데이터 출처 비공개: 사전학습 코퍼스 출처가 공개되지 않아 저작권·라이선스 측면 투명성은 낮다. 공공기관 도입 시 정보보호 심의 단계에서 쟁점화 가능.

지정학적 고려: Alibaba Cloud(중국) 출처 모델로 일부 공공기관은 보안 정책상 사용 제약 가능성. 단 모델 가중치 자체는 오픈소스이며 추론은 완전 오프라인(인터넷 차단 망분리 환경)에서 수행 가능 — 외부 데이터 송신 경로 없음.

라이선스: Apache-2.0이라 라이선스 자체 제약은 없음. EXAONE 같은 비상업 한정 모델 대비 자유도 우위.

성능 한계: 70B+ 모델 대비 복잡한 추론·다단계 reasoning은 약함. GPT-4o급 응답 품질 기대는 부적절 — 보조 도구 포지션 명확화 필요.

장문 처리: 32K 네이티브를 넘는 입력 시 YaRN rope_scaling 설정 필요. 짧은 입력에서 YaRN 활성 시 오히려 성능 저하 가능 — 입력 길이별 분기 라우팅 권장.

08종합 의견

광진구 단위에서 "외부 API 호출 없이 내부 망에서 가동 가능한 한국어 보조 모델"이 필요하다는 전제에서 Qwen2.5-7B-Instruct는 POC 1차 후보로 적합하다. 단일 GPU(RTX 4090 24GB) 또는 4-bit 양자화 시 16GB GPU 1장에서 vLLM 기반 OpenAI 호환 API 서빙이 가능하여 기존 ChatGPT 연동 코드를 endpoint만 교체해 시범 운영할 수 있다.

다만 도입 전 다음 5가지 검증이 필요하다 — (a) 광진구 자체 민원 샘플 200~500건에 대한 응답 품질 블라인드 비교(GPT-4o · EXAONE-3.5 · Kanana-nano 등 후보군과), (b) RAG 파이프라인 구축 후 출처 인용 정확도 측정, (c) 한국어 격식·존댓말 적합성 휴먼 평가, (d) 개인정보 포함 입력에 대한 PII 마스킹 정책 적용 후 응답 적정성 확인, (e) 중국 출처 모델 가중치 사용에 대한 보안성 검토 및 정보보호 심의.

행정문서 도메인 적응이 필요하면 구청 공개 데이터(조례·민원 사례·보도자료) 5,000~10,000 샘플로 LoRA SFT 1~2 epoch 수준의 경량 튜닝으로도 도메인 정확도 개선이 가능하다. 비상업 한국어 특화 모델(EXAONE-3.5-2.4B/7.8B, Kanana 시리즈)을 병행 평가하여 최종 채택 모델은 한국어 품질 + 라이선스 + 운영 비용 + 보안성 4축에서 결정할 것을 권한다.
