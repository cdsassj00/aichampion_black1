01모델 선택 이유

방위사업 분야의 계약 문서(제안요청서·제안서·계약일반조건·계약특수조건·세부품목내역) 5종과 관련 규정(방위사업관리규정·수정계약지침·군수품조달관리규정·국가계약법) 4종을 대상으로 한 한국어 RAG(Retrieval-Augmented Generation) 검색·질의응답 서비스의 베이스 모델로 Qwen2.5-7B-Instruct를 선택하였다. 선택 근거는 네 가지이다. 첫째, 약 18조(18T) 토큰 규모의 다국어 코퍼스로 사전학습된 모델로 한국어를 포함한 29개 언어를 지원하며, 한국어 행정·법률 도메인 fine-tuning 베이스로 가장 많이 채택되는 7B급 오픈 모델이다. 둘째, 32K 네이티브 컨텍스트(YaRN 확장 시 최대 128K)로 방위사업관리규정·국가계약법처럼 조항 수가 많은 법령을 RAG 입력에 통째로 포함할 수 있고, 다중 문서 검색 결과를 동시에 처리할 수 있다. 셋째, Apache-2.0 라이선스로 정부·공공·상업 활용에 모두 자유로워 PoC 단계에서 본 도입까지 라이선스 협의 부담이 없다. 넷째, 7B 사이즈는 RTX 4090 24GB 단일 GPU에서 추론 가능하여 부서 서버 단계의 온프레미스 PoC에 적합하고, 검색된 컨텍스트를 충실히 따라가는 instruction following 능력이 동급 7B 모델 중 최상위 수준으로 평가된다.

03학습 데이터와 튜닝 방식

사전학습 데이터는 약 18조(18T) 토큰 규모의 대규모 다국어·다도메인 코퍼스로, 웹 텍스트·코드·수학·다국어 텍스트가 포함된 것으로 공개되었다. 한국어를 포함한 29개 언어를 지원하지만 도메인별·소스별 비중과 구체 데이터셋 구성은 모델 카드 공개 범위 내에서 부분만 확인되며, 산업·법률 등 특정 도메인 데이터의 별도 포함 여부는 명시되어 있지 않아 "공개 범위 내에서는 확인 어려움"으로 표기한다. 튜닝 방식은 베이스 모델(Qwen2.5-7B) 사전학습 이후 Instruct 버전 생성을 위해 SFT(Supervised Fine-Tuning)와 DPO(Direct Preference Optimization)를 순차 적용한 것으로 공개되어 있으며, RLHF의 별도 적용 여부는 모델 카드에 명시되어 있지 않다. 컨텍스트 길이 확장은 RoPE(theta = 1,000,000) 기반으로 32K 네이티브를 제공하고, YaRN 적용 시 최대 128K까지 확장 가능하다.

04파라미터 및 구조적 특징

ArchitectureQwen2 (Qwen2ForCausalLM, model_type: qwen2) — Llama 계열 Decoder-only Transformer + RoPE + RMSNorm + SwiGLU + GQA
Context Length32,768 토큰 네이티브 (max_position_embeddings) · 131,072 토큰 sliding window 확장 (YaRN)
Hidden Size3,584 (intermediate_size 18,944, SwiGLU FFN)
Layers28 (num_hidden_layers)
Attention Heads28 query heads · 4 key-value heads (GQA, Grouped Query Attention 7:1)
TokenizerByte-level BPE (Qwen tokenizer) · vocab size 152,064 · ChatML 특수 토큰 (<|im_start|>, <|im_end|> 등)

05모델 파일 구성

Hugging Face 저장소(Qwen/Qwen2.5-7B-Instruct)의 주요 파일 역할은 다음과 같다. (1) config.json — 모델 아키텍처와 파라미터를 정의하는 핵심 파일로, hidden_size·num_hidden_layers·num_attention_heads·max_position_embeddings 등 04번 섹션의 모든 구조 정보가 여기서 추출되며 추론·재학습 시 가장 먼저 로드된다. (2) tokenizer.json — fast tokenizer 본체로 BPE 병합 규칙과 어휘를 포함한다. (3) tokenizer_config.json — 토크나이저 설정과 ChatML 특수 토큰(<|im_start|>, <|im_end|> 등)의 매핑을 정의하여 대화 포맷을 결정한다. (4) vocab.json + merges.txt — BPE 어휘 사전과 병합 규칙(legacy 포맷, slow tokenizer 호환). (5) special_tokens_map.json — BOS·EOS·PAD 등 특수 토큰의 정식 매핑. (6) model-0000X-of-00004.safetensors (4분할) + model.safetensors.index.json — 모델 가중치(bfloat16 약 15GB)와 분할 파일 인덱스로 분산 로딩에 사용된다. (7) generation_config.json — 추론 시 기본 생성 파라미터(temperature 0.7, top_p 0.8, top_k 20, repetition_penalty 1.05, do_sample true)가 정의된다.

06활용 가능 업무

방위사업 분야 계약·규정 RAG 검색 서비스의 다섯 가지 구체 시나리오를 상정한다. 첫째, 계약 조항 즉시 인용 — "지체상금 면제 사유는 무엇인가" 질의 시 계약일반조건과 국가계약법의 해당 조항을 출처(문서명·조항번호·페이지)와 함께 자연어로 응답한다. 둘째, 규정 간 차이 비교 — "수정계약 절차가 방위사업관리규정과 수정계약지침에서 어떻게 다른가" 질의에 두 규정의 공통점·차이점·우선 적용 관계를 정리하여 응답한다. 셋째, 사업별 특수조항 추출 — 계약특수조건에서 본 사업에 한정된 조건만 자동 분리·요약하여 일반조건과의 충돌·우선 적용 여부를 안내한다. 넷째, 세부품목 단가 근거 확인 — 특정 품목의 단가 책정 근거 조항과 과거 단가를 추적하여 산정 근거를 제시한다. 다섯째, 신규 사업 기획 시 선례 검색 — 과거 제안요청서에서 유사 사업의 평가 기준·자격 요건·특수 조건을 자동 검색·인용한다. 활용 형태는 부서 내부 챗봇 UI에 출처 명시(문서명·조항·페이지)와 원문 인용을 함께 제시하는 폐쇄형 시범 운영을 가정한다.

07한계와 주의사항

여섯 가지 한계와 주의사항이 있다. 첫째, 환각(hallucination) 가능성 — 단서·예외 조항이나 신구조문 비교에서 잘못된 인용이 발생할 경우 계약 분쟁·감사 리스크로 직결되므로, RAG 인용 표시와 사람의 최종 검수가 절대적으로 필요하다. 둘째, 한국어 정밀도 — 일반 행정 문서 처리는 우수하나 법령 전문 용어나 약어(예: 「국계법」 등 약식 인용)에서 한국 토종 모델(EXAONE 등)에 비해 미세하게 약할 수 있다. 셋째, 표·숫자 처리의 약점 — 세부품목내역과 같이 표·숫자 위주의 정형 문서는 LLM 일반의 약점이며, 별도의 표 추출 파이프라인이나 vision 모델 보조가 필요하다. 넷째, 평가 데이터 부재 — 한국 방산·계약 도메인에 한정된 공개 평가 데이터셋이 사실상 없어 자체 QA 평가셋 50~100쌍의 자체 구성이 필수이다. 다섯째, 라이선스 측면에서 Apache-2.0으로 상업·정부 활용은 자유로우나, 모델 출처(Alibaba Cloud)는 운영 환경 본 도입 단계에서 별도의 보안·감사 검토 대상이 될 수 있다. 여섯째, 컨텍스트 한계 — 네이티브 32K 초과 시 YaRN 활성화가 필요하며, 매우 긴 다문서 입력 시 정확도 저하(중간 손실)가 발생할 수 있다.

08종합 의견

Qwen2.5-7B-Instruct는 방위사업 계약·규정 RAG 검색 서비스의 PoC 베이스 모델로 적합하다고 판단한다. 도입을 위한 단계별 권장 조건은 다음과 같다. (1) 운영 환경 — 부서급 온프레미스 서버(RTX 4090 24GB 이상)에 RAG 챗봇 형태로 배포한다. (2) 1차 PoC — fine-tuning 없는 RAG-only 구성으로 시작하여 OpenSearch 또는 ChromaDB 기반 벡터 검색 + 한국어 임베딩(ko-sroberta 또는 BGE-M3) + Qwen2.5-7B-Instruct로 파이프라인을 구축한다. 사실 인용은 RAG가, 톤·서술은 LLM이 담당하는 분리 원칙을 유지한다. (3) 평가 — 자체 QA 50~100쌍을 도메인 실 질문 기반으로 구성하고 BLEU·LLM-as-a-Judge·인용 정확도(human-eval) 3축으로 측정한다. (4) 점수 부족 영역 보정 — 한국어 톤·법률 용어가 부족할 경우에 한해 LoRA fine-tuning으로 도메인 보정을 추가하되, 사실 인용은 RAG의 책임으로 유지하여 환각 위험을 통제한다. (5) 비교 PoC — 동급 후보 모델(EXAONE-3.5-2.4B-Instruct, Kanana-Nano-2.1B-Instruct, EXAONE-3.5-7.8B-Instruct)과 동일 평가셋으로 비교 PoC를 1회 추가 수행하여 한국어 도메인 정밀도와의 trade-off를 정량 검증한다. (6) 본 도입 조건 — 인용 정확도 90% 이상, 환각률 5% 이하, 사용자 만족도 4.0/5 이상이 입증된 후 보안팀 검토(모델 가중치 무결성, 학습 데이터 출처, 망 분리 운영성) 단계로 진입한다.
