01모델 선택 이유

본인은 현재 「AI 지방세 지식인」(olta.re.kr 공개자료를 기반으로 지방세 담당 공무원의 법령·유권해석 질의에 즉시 응답하는 RAG 기반 Q&A 시스템) Phase 3 단계에서 폐쇄망 LLM 전환을 준비 중이며, 본 보고서는 그 후보 SLM 1차 스크리닝 결과이다. Gemma-4-E2B-it를 선정한 사유는 다음 다섯 가지이다.

첫째, 규모 적합성 — Effective 2B(임베딩 포함 5.1B)라는 경량 구조는 1B~7B 검토 기준 내에 정확히 위치하며, 고령군과 같은 기초지자체가 자체 GPU 서버를 갖추기 어려운 현실에서 단일 GPU(예: NVIDIA L4, A10) 또는 고성능 워크스테이션급 자원으로도 추론이 가능하다.

둘째, Apache-2.0 라이선스 — 공공기관에서 가장 우려하는 라이선스 분쟁 위험이 사실상 없는 허용형 오픈소스 라이선스이며, 상업적·내부 행정용 활용에 제약이 없다. Llama 3 Community License 계열의 사용자 수 임계 조항도 적용되지 않는다.

셋째, 128K 장문 컨텍스트 — 지방세법·지방세특례제한법·시행령·시행규칙·유권해석을 한 번에 컨텍스트로 주입할 수 있는 수준이며, 조문 단위 청킹을 통해 RAG로 압축된 근거 문서(Top-k 10~20개)를 충분히 수용한다.

넷째, Native Function Calling 지원 — 「AI 지방세 지식인」이 단순 RAG에서 MCP(Model Context Protocol) 구조로 진화할 때 Tool Use 안정성이 결정적이며, Gemma 4는 이를 사전학습 단계에서 네이티브 지원한다.

다섯째, Thinking Mode와 Instruction-Tuned 변형 — `<|think|>` 토큰 기반 사고 모드를 통해 복잡한 세법 적용(예: 공유지분 상속 취득세 산정) 시 단계별 추론을 수행할 수 있으며, `-it` 접미사가 붙은 instruction-tuned 모델로서 별도 SFT 없이 즉시 한국어 질의응답에 활용 가능하다.

03학습 데이터와 튜닝 방식

1. 사전학습(Pre-training) 데이터
모델 카드에서 공개한 학습 데이터 구성은 ① Web Documents(140개 이상 언어를 포함한 광범위 웹 텍스트), ② Code(다국어 프로그래밍 코드), ③ Mathematics(수학·논리 추론 텍스트), ④ Images(다양한 시각 데이터)이며, 지식 컷오프(Cutoff)는 2025년 1월이다. 이는 2025년 개정 지방세법(부동산·차량 취득세 관련 일부 조문)이 부분적으로만 반영되었을 수 있음을 의미하며, RAG 파이프라인에서 시행일자 메타데이터 필터링이 반드시 필요한 근거가 된다. 다국어 사전학습은 35개 이상 언어에 즉시 활용 가능하나 한국어 비중은 별도 공개되지 않아 공개 범위 내에서는 정량 확인이 어렵다.

2. 데이터 전처리
CSAM(아동 성착취물) 다단계 필터링, 자동화된 개인정보·민감정보 필터링, 콘텐츠 품질·안전성 기반 추가 필터링이 적용되었다. 공공기관 도입 관점에서 학습 데이터 안전성에 대한 1차 검증은 충분히 수행된 것으로 판단된다.

3. 사후학습(Post-training)·튜닝 방식
`-it` 변형은 Instruction-Tuning이 완료된 버전으로, Google이 발표한 별도 블로그·기술 문서에 의하면 SFT(Supervised Fine-Tuning)와 안전성 강화를 위한 인간 평가 기반 튜닝이 결합된 형태이다. 다만 RLHF·DPO·LoRA의 구체 적용 여부와 비율은 모델 카드에 명시적으로 공개되지 않는다(Google의 일반 정책상 사후학습 세부 절차는 비공개). 안전성 평가에서 Gemma 3 대비 거의 모든 안전 카테고리에서 의미 있는 개선을 보였다고 자체 보고하고 있으며, 별도 안전 필터 미적용 상태에서도 정책 위반율이 낮다고 평가된다.

4. 한국어·법률 도메인 특화 여부
한국어 법령·행정 도메인 특화 학습은 적용되지 않았다. 따라서 본 모델을 「AI 지방세 지식인」 폐쇄망 환경에 적용하려면 (1) 지방세 법령·유권해석 코퍼스를 활용한 LoRA/QLoRA 기반 도메인 적응(Domain Adaptation), 또는 (2) 강력한 RAG 파이프라인으로 외부 지식을 주입하는 방식이 필요하다. 본 시스템에서는 RAG 우선, 필요 시 LoRA 추가 적용 전략이 합리적이다.

04파라미터 및 구조적 특징

ArchitectureGemma4ForConditionalGeneration (model_type: gemma4) — 텍스트 디코더 + 비전 인코더(gemma4_vision) + 오디오 인코더(gemma4_audio) 결합 멀티모달. 텍스트는 Decoder-only Transformer, Hybrid Attention(슬라이딩 4 + 글로벌 1 반복, 35 레이어 중 7개가 full_attention), 극단적 GQA(8 Q-heads → 1 KV-head, 8:1 비율), Per-Layer Embeddings(PLE), Proportional RoPE(글로벌, partial_rotary_factor=0.25, theta=1e6) + 표준 RoPE(슬라이딩, theta=1e4), use_double_wide_mlp=true, tie_word_embeddings=true(입출력 임베딩 공유), final_logit_softcapping=30.0
Context Lengthmax_position_embeddings=131072 (=128K 토큰) · sliding_window=512 · 35개 레이어 중 7개(레이어 4·9·14·19·24·29·34)가 글로벌 어텐션, 나머지 28개는 512 토큰 슬라이딩 윈도우 → 5중 1(슬라이딩 4 + 글로벌 1) 패턴 반복. 마지막 레이어는 항상 글로벌
Hidden Size텍스트 디코더 hidden_size = 1536 · intermediate_size = 6144(FFN, double_wide_mlp) · hidden_size_per_layer_input = 256(PLE 차원) · vocab_size_per_layer_input = 262,144 (참고 — 비전 hidden_size 768, 오디오 hidden_size 1024)
Layers텍스트 디코더 num_hidden_layers = 35 · num_kv_shared_layers = 20 (35개 중 20개 레이어가 KV cache를 공유하여 메모리 절약) · 비전 인코더 16 레이어 · 오디오 인코더 12 레이어
Attention Headsnum_attention_heads = 8 / num_key_value_heads = 1 (8:1 GQA, 사실상 MQA에 가까운 극단적 KV 압축) · head_dim = 256(슬라이딩) · global_head_dim = 512(글로벌) — Q 차원 = 8×256 = 2048, KV는 글로벌 시 512차원 단일 헤드. 본 비대칭 head_dim이 외부 추론 엔진(ONNX Runtime GenAI 등) 호환의 핵심 변수
TokenizerSentencePiece 기반 · vocab_size = 262,144(262K) · 140+ 다국어 토큰화(한국어 포함) · 멀티모달 특수 토큰 정의(image_token_id=258880, boi/eoi=255999/258882, audio_token_id=258881, boa/eoa=256000/258883, video_token_id=258884) · BOS=2, EOS=1·106(채팅 종료), PAD=0 · Gemma 3 호환 system/user/assistant 채팅 템플릿

05모델 파일 구성

Hugging Face 저장소(google/gemma-4-E2B-it)는 BF16 정밀도 safetensors 포맷으로 배포되며, Transformers 라이브러리와 직접 호환된다. 주요 파일 구성은 다음과 같다.

① config.json — 모델 아키텍처 정의 파일. 본 모델은 단일 Gemma4ForConditionalGeneration 클래스 아래 3개의 서브 컨피그가 중첩된 구조이다. 
▶ text_config(model_type=gemma4_text): hidden_size=1536, intermediate_size=6144, num_hidden_layers=35, num_attention_heads=8, num_key_value_heads=1, head_dim=256/global_head_dim=512, sliding_window=512, max_position_embeddings=131072, vocab_size=262144, num_kv_shared_layers=20, layer_types 배열로 35개 레이어의 sliding/full 어텐션 패턴 명시. 
▶ vision_config(model_type=gemma4_vision): hidden_size=768, num_hidden_layers=16, patch_size=16, vision_soft_tokens_per_image=280. 
▶ audio_config(model_type=gemma4_audio): hidden_size=1024, num_hidden_layers=12, conv_kernel_size=5(서브샘플링 컨볼루션), output_proj_dims=1536(텍스트 디코더와 정렬). 폐쇄망 환경 배포 시 KV cache 메모리 산정의 근거가 되는 핵심 파일이며, 특히 num_kv_shared_layers=20과 num_key_value_heads=1이 메모리 효율의 핵심 변수이다.

② tokenizer.json / tokenizer_config.json / special_tokens_map.json — SentencePiece 기반 토크나이저 정의. 262K 어휘 사전과 함께 Gemma 4 채팅 템플릿(<|turn>system / user / model), 사고 모드 제어 토큰(<|think|>, <|channel>thought), 멀티모달 입력 마커가 정의되어 있다. 한국어 토크나이징 효율은 vocab 크기가 큰 만큼 영문 대비 토큰 인플레이션이 비교적 적을 것으로 기대된다.

③ model.safetensors (분할 저장 가능) — 약 5B 파라미터의 BF16 가중치 파일. 디스크 사용량은 약 9.3GB(BF16 기준), 4bit 양자화(IQ4_XS) 시 약 3GB 수준. Pickle 보안 위험이 없는 safetensors 포맷이라 폐쇄망 배포에 적합하다.

④ generation_config.json — 추론 시 기본 샘플링 파라미터. 권장 설정은 temperature=1.0, top_p=0.95, top_k=64이다.

⑤ preprocessor_config.json / processor_config.json — 멀티모달 입력 전처리 설정(이미지 토큰 예산 70/140/280/560/1120, 오디오 30초 제한 등).

⑥ chat_template.jinja — 멀티턴 대화 템플릿. 사고 모드 활성·비활성 분기, 이전 턴의 thought 블록 제거 규칙이 포함된다.

배포 형태별로는 본 원본(BF16 safetensors) 외에 GGUF(llama.cpp용, ggml-org/gemma-4-E2B-it-GGUF), LiteRT-LM(모바일·온디바이스용), ONNX(Transformers.js)가 별도 저장소로 제공되어 폐쇄망 인프라 형태에 따라 선택적 활용이 가능하다.

06활용 가능 업무

본 모델은 「AI 지방세 지식인」 Phase 3(폐쇄망 LLM 전환) 환경에서 다음 다섯 영역에 우선 적용 가능하다.

① 지방세 법령 RAG 질의응답(핵심 시나리오) — 지방세 담당 공무원의 1차 질의(예: "공유지분 상속 차량의 취득세 산정 방법", "지방세특례제한법 제○조 감면 적용 요건")에 대해, 사전 구축된 olta.re.kr 코퍼스(지방세법·시행령·시행규칙·유권해석·심판례) Top-k 검색 결과를 컨텍스트로 주입하여 근거 인용형 답변을 생성한다. 128K 컨텍스트 덕분에 조문 단위 청킹 결과 10~20개를 충분히 수용한다.

② 2단계 LLM 검증 파이프라인의 1차 답변 생성기 — 본 시스템의 차별화 요소인 "1차 LLM 답변 생성 → 2차 LLM 근거 대조 검증" 구조에서, 본 모델은 1차 답변 생성기 역할을 수행한다. Thinking Mode 활성화 시 단계별 추론(예: 상속 유형 분류 → 공유지분 계산 → 감면 해당 여부 → 최종 세액)을 수행하여 답변의 논리적 일관성을 높일 수 있다.

③ 민원 문서 자동 요약 및 분류 — 시민이 작성한 장문 민원(불복 청구서, 이의신청서 등)을 입력받아 6대 분류(법령 적용·세액 산정·감면 적용·절차 문의·불복·기타) 중 자동 라벨링 및 핵심 쟁점 3줄 요약을 생성한다.

④ 내부 회의록·보고서 초안 작성 — 폐쇄망 환경에서 내부 회의 음성을 STT로 변환한 후(또는 본 모델의 ASR 기능 직접 활용 — 30초 단위 분할), 회의록 정형화·핵심 결정사항 추출·후속 조치 To-Do 추출에 활용한다. 다만 ASR 한국어 정확도는 별도 검증이 필요하다.

⑤ 함수 호출(Function Calling) 기반 MCP 도구 호출 에이전트 — 본인이 향후 진화시킬 MCP 구조에서 LLM이 "지방세법 제○조 조회" 등 도구를 명시적으로 호출하도록 설계할 때, Native Function Calling 지원이 핵심이다. 답변 근거가 자동으로 로그에 남아 공공 AI 감사 추적성 요구를 구조적으로 충족한다.

⑥ (확장) 멀티모달 활용 — 차량 등록증·납세증명서·매매계약서 이미지를 OCR + 의미 추출하여 취득세 산정 시 자동 입력 가능. 다만 한국어 OCR 정확도와 행정문서 양식 인식률은 별도 PoC 검증이 필요하다.

07한계와 주의사항

① 한국어 법률 도메인 성능의 불확실성 — 사전학습 데이터의 한국어 비중·법률 문서 비중이 비공개이며, 모델 카드에 한국어 단독 벤치마크가 제시되지 않는다. MMMLU(다국어) 67.4%는 일반 지식 기준이며, 지방세법 같은 전문 법률 영역에서의 정확도는 별도 PoC 검증 없이는 신뢰할 수 없다. 환각(특히 조문 번호·금액·요건 등의 사실 오류) 가능성을 RAG 근거 강제 인용 + 2차 검증 LLM으로 차단해야 한다.

② 일반 벤치마크상 절대 성능의 제약 — Gemma 4 E4B 대비 모든 항목에서 명확한 성능 차이가 있다. MMLU Pro 60.0%(E4B 69.4%), AIME 2026 37.5%(E4B 42.5%), LiveCodeBench v6 44.0%(E4B 52.0%), GPQA Diamond 43.4%(E4B 58.6%). 즉 추론·수학·코딩이 동시에 요구되는 복합 민원에는 E4B가 더 적합할 수 있다. 본 모델은 단순 RAG 응답에 우선 적용하고, 복합 추론은 상위 변형으로 라우팅하는 하이브리드 구조가 합리적이다.

③ 학습 데이터 컷오프(2025년 1월) — 2025년 중반 이후 개정된 지방세 관련 법령은 모델 내부 지식에 반영되지 않았다. 따라서 RAG 시 시행일자 메타데이터 기반 최신 조문 우선 검색이 필수이며, 단독 활용 시 구 조문 답변 위험이 존재한다.

④ 일반적 한계(모델 카드 명시) — 학습 데이터 편향, 사실 정확성 한계(지식 베이스 아님), 미묘한 언어 뉘앙스·반어법·은유 처리의 어려움, 상식 추론 한계 등이 명시되어 있다. 공공 행정에 적용 시 "참고 의견" 포지션을 유지해야 하며, 최종 판단은 반드시 담당 공무원이 수행하는 책임 구조를 코드와 UI에 강제 노출해야 한다.

⑤ 라이선스 측면 — Apache-2.0이지만 Google이 별도로 정의한 "Gemma Prohibited Use Policy" 준수 의무가 있다. 위법행위 보조, 차별·증오 콘텐츠 생성 등 일반적 금지 행위 외 공공기관 활용에 직접 충돌하는 조항은 확인되지 않으나, 도입 직전 법무 검토는 권장된다.

⑥ 폐쇄망 배포 시 추가 고려사항 — Hugging Face에서 모델을 다운로드하려면 Google Gemma Use Policy 동의가 필요하며, 폐쇄망 내부 반입 시 (1) 가중치 무결성 검증, (2) 사고 모드 토큰 처리 라이브러리 버전 일치, (3) tokenizer.json의 bos/eos/pad/unk 토큰 동일성 확인이 운영상 중요하다.

⑦ 극단적 KV 압축의 양면성 — config.json 분석 결과 num_key_value_heads=1(MQA에 가까운 8:1 GQA)과 num_kv_shared_layers=20(35개 중 20개 레이어 KV cache 공유)이 적용되어 있다. 장점은 128K 컨텍스트에서도 KV cache 메모리가 동급 모델 대비 현저히 작다는 점이나(엣지 디바이스 친화), 단점은 외부 추론 엔진(ONNX Runtime GenAI, vLLM 일부 버전, TensorRT-LLM)이 비대칭 head_dim(슬라이딩 256 / 글로벌 512)과 KV 공유 패턴을 아직 완전히 지원하지 않아 llama.cpp 외 엔진에서 변환·로딩 실패가 보고된다(Microsoft onnxruntime-genai Issue #2062 등). 폐쇄망 1차 배포는 Transformers + vLLM 최신판 또는 llama.cpp(GGUF) 조합이 가장 안전하다.

08종합 의견

결론적으로 Gemma-4-E2B-it는 「AI 지방세 지식인」 Phase 3 폐쇄망 환경의 1차 테스트 모델로 적합하다. 5B 파라미터급에서 128K 장문 컨텍스트, Native Function Calling, Apache-2.0 라이선스, BF16 safetensors 표준 포맷이라는 4대 조건을 동시에 만족하는 SLM은 현 시점에 흔치 않으며, 단일 GPU 인프라가 현실적인 기초지자체 환경에 부합한다.

다만 다음 3단계 추가 검증을 거친 후 실서비스 적용을 결정할 것을 제안한다.

[PoC 1단계 — 한국어 법률 응답 정성 평가] 한국지방세연구원 공개 사례 50건을 골든셋으로 구성하여, 본 모델 단독 답변 vs 본 모델 + RAG vs Qwen2.5-7B-Instruct + RAG vs LG EXAONE 3.5(7.8B) + RAG의 4-way 비교 평가를 실시한다. 평가 항목은 ① 조문 인용 정확도, ② 사실 오류율, ③ 답변 형식 준수율(①~⑥ 구조), ④ 지방세 전문 용어 적합성 4개. 담당 공무원 3인 블라인드 평가로 통계 검증.

[PoC 2단계 — 실 부하 환경 추론 성능] 단일 NVIDIA L4 또는 A10 GPU에서 동시 사용자 5~10명, 평균 입력 4K 토큰, 평균 출력 1K 토큰 조건으로 TTFT(Time-to-First-Token), TPS(Tokens-per-Second), 메모리 사용량을 측정한다. 4bit 양자화(Q4_K_M GGUF 또는 IQ4_XS)와 BF16 원본의 품질 차이를 동일 골든셋으로 비교 검증한다.

[PoC 3단계 — 한국어 LoRA 도메인 적응 가능성] olta.re.kr 누적 Q&A 5,000건과 지방세법 조문 코퍼스를 활용한 QLoRA 미세조정을 실시하여, 도메인 적응 후 답변 품질이 RAG 단독 대비 유의미하게 개선되는지 평가한다. 비용 대비 효과가 입증되면 Phase 3 정식 배포 모델 후보로 확정한다.

[대안 검토] 만약 한국어 정확도가 EXAONE 3.5 / Qwen2.5 / HyperCLOVA X Seed 대비 명확히 떨어진다면, 본 모델은 (a) 1차 답변 생성기, (b) 보조 분류·요약 모델로 역할을 한정하고, 메인 추론은 한국어 특화 모델로 전환하는 듀얼 모델 아키텍처가 합리적이다. 본 보고서는 본 모델을 단독 솔루션이 아닌 "폐쇄망 후보군의 강력한 비교 기준선(Strong Baseline)"으로 자리매김할 것을 제안한다.
