01모델 선택 이유

저는 현재 정보화 부서에서 전산장비 관리자로서 기존 인프라(HP DL580 Gen9 5대, Rocky Linux 10)를 활용한 AX 기반 환경 조성업무를 자발적으로 추진 하고있습니다. 
신규 고가의 GPU 도입이 어려운 환경을 고려하여, nvidia T4 16GB 그래픽카드를 해당 서버에 설치하여 테스트를 진행했으며 Ollama API를 통한 챗봇 PoC에 성공했습니다.
현 사양에서는 현 서버에 맞는 4B급 경량 모델 운용으로 AX 전한에 가장 현실적인 모델로 gemma-3-4b를 선정하였습니다. 
1.  효율적 리소스: 낮은 자원으로도 전직원을 대상으로 동시접속자 10명 이용가능 확인 
2. 업무 적합성: 행정 문서 요약, 민원 초안 작성 등 텍스트 업무에도 활용가능
3. 기술 신뢰도: 구글 딥마인드(Gemma) 모델로 gemma4 등 모델 발전에 따라 서비스 업데이트 가능
향후 서버 5대를 AX 전환의 마중물로  파싱, 임베딩, 챗봇, 보고서 작성등 맞춤형 서비스에 맞는 모델을 적용할 계획입니다.

03학습 데이터와 튜닝 방식

Gemma 3 4B 모델은 약 4T 토큰 규모 데이터로 학습되었고 웹 문서·코드·수학 데이터와 이미지 데이터가 포함 하였습니다. 140개 이상 언어를 다루도록 구성되었으며, 안전성과 품질 확보를 위해 민감정보/유해정보 필터링이 적용되었습니다.
또한  IT(Instruction-tuned) 버전으로, 사전학습 이후 대화·지시 이행 성능을 높이는 후속 튜닝이 반영된 버전입니다.  기술 보고서에서는 distillation 기반 학습과 RL 기반 post-training 개선이 언급됩니다. 다만 세부 데이터셋 목록(기관별 비율·원천 URL 등)과 모든 튜닝 하이퍼파라미터는 공개 범위 내에서 확인이 제한적입니다.

04파라미터 및 구조적 특징

ArchitectureGemma3ForConditionalGeneration
(텍스트 디코더 + SigLIP 기반 비전 인코더 결합)
Context Length입력 최대 128K(131072) 토큰, 출력 최대 8192 토큰 (4B 기준)
Hidden Size텍스트 hidden_size 2560 (vision hidden_size 1152)
Layers텍스트 34층, 비전 27층
Attention Heads텍스트 8 heads / 4 KV heads (GQA), 비전 16 heads
TokenizerSentencePiece 계열 토크나이저, vocab size 262,208

05모델 파일 구성

gemma-3-4b-it는 "구조 + 가중치 + 토크나이저 + 전처리/생성 규칙" 로 구성됩니다. 
1) 구조/런타임 설정 계층
- config.json: 모델 클래스 선택, 텍스트/비전 모듈 구조, hidden size, layer 수, attention heads, rope 설정 등 핵심 하이퍼파라미터가 들어 있습니다. 서버에서 이 파일을 먼저 읽고 메모리 할당 크기와 연산 그래프를 결정합니다.
- generation_config.json: max_new_tokens, eos_token_id, temperature, top_p 등 기본 생성 정책이 포함되어 있습니다. 서비스 코드에서 별도 파라미터를 주지 않으면 이 기본값이 적용되어 출력 길이/응답 성향이 달라집니다.
2) 가중치(모델 본체) 계층
- model-00001-of-0000N.safetensors 다중 파일: 실제 파라미터 텐서가 샤딩되어 저장됩니다. 단일 대용량 파일보다 로딩 안정성과 배포 편의성이 높습니다.
- model.safetensors.index.json: 각 텐서 이름이 어느 샤드 파일에 들어있는지 인덱스를 제공합니다. 런타임은 이 인덱스를 기준으로 필요한 텐서를 순차/병렬 로딩합니다.
3) 토크나이저 계층
- tokenizer.model (SentencePiece) + tokenizer.json: 한국어/영어/숫자/기호를 토큰 단위로 분해하는 실제 규칙과 vocab 매핑입니다. 동일 문장이라도 토크나이저가 다르면 토큰 수와 추론 비용이 달라집니다.
- tokenizer_config.json, special_tokens_map.json, added_tokens.json: BOS/EOS/PAD, 역할 토큰, 채팅 제어 토큰 정의가 포함됩니다. 누락되면 프롬프트 포맷이 깨져 성능 저하나 출력 이상이 발생할 수 있습니다.
4) 멀티모달 전처리 계층
- preprocessor_config.json, processor_config.json: 이미지 리사이즈/정규화, 텍스트-이미지 입력 결합 방식, processor 호출 순서를 규정합니다.
- chat_template.json: system/user/assistant 메시지를 모델이 기대하는 문자열 포맷으로 직렬화합니다. 챗봇 품질은 이 템플릿 일관성의 영향을 크게 받습니다.
운영 관점 체크포인트: 실제 서비스 배포 시에는 파일 버전 불일치(예: config는 최신인데 tokenizer는 구버전)만 발생해도 응답 품질 저하·로딩 오류가 생길 수 있으므로, 모델 리비전 고정과 체크섬 검증 절차를 함께 두는 것이 안전합니다.

06활용 가능 업무

업무를 하면서 확인된 내부정보를 인공지능이 관리하고 알려주는 "메모마인드" PoC 진행 중
(운영방식) 
- 리눅스 서버 활용 Ollama Serve API 인공지능 모델 활용 
- 메모 또는 내부 정보에 대한 자료관리(Json), 메모 내용에 대한 Rag 방식의 답변
(관리자 활용 방안) 
- 예산편성시 담당자에게 요청하는 자료 및 보고에 대한 자료 관리, 사업 추진현황 관리
- gemma3:4B 수준에서 업무에 대한 업무의 기본 정보 활용
(실무 공무원 활용) 
- 업무 추진시 획득한 정보에 대한 메모 관리, 인사인동에 따른 업무인수 인계 AI 관리
- 내부정보를 이용한 AI 활용

07한계와 주의사항

1. 복합 추론·최신 사실성·도메인 초정밀 답변에서 상위 대형 모델 대비 한계
2. 환각(hallucination) 가능성이 존재하여 활용방식에 대한 교육 필요

08종합 의견

gemma-3-4b와 같은 경량 모델을 채택함으로써, 기존 서버 자원을 극대화하는 동시에 단계적인 AI 서비스 확대를 실현할 수 있는 최적의 방법입니다. 이는 실무 현장에 즉시 투입 가능한 맞춤형 행정 서비스 개발과 병행되어, 예산 효율성을 확보함과 동시에 지방정부의 성공적인 AX(인공지능 전환) 선도 사례가 될 것으로 기대됩니다
